# aion-governance vs three-role prototype v0.1

## Purpose
Compare the historical `aion-governance` system with the newer three-role AI-only prototype, identify recorded hard requirements from the historical version, and recommend a migration path suitable for AION's current phase.

---

## 1. Executive judgment

The two systems should not be treated as rivals.

- `aion-governance` = historical governance mother-source
- `three-role prototype` = current AI-only operating-system kernel

Best path:
- preserve hard requirements from `aion-governance`
- compress them into the lighter three-role world
- treat the new prototype as the active forward-looking model

---

## 2. Historical hard requirements found in `aion-governance`

### 2.1 GitHub is the formal governance center
Recorded rule:
- Discord may start and drive action
- GitHub is where formal governance takes effect

Interpretation:
- chat is not the long-term truth layer
- governance, audit, closure, and archival need a durable record center

### 2.2 Discord is for short routing, not full execution text
Recorded rule:
- Discord should carry short commands, status updates, and routing only
- long execution text belongs in GitHub

Interpretation:
- this protects the Monarch from transport burden
- this reduces fragmented governance state

### 2.3 The Monarch must not become the manual relay layer
Recorded rule:
- the Monarch is not a manual relay or monitor layer
- automated submission pipelines should handle content transport

Interpretation:
- manual copy-paste is organizational immaturity
- automation should replace relay burden

### 2.4 AI-first document writing and minimal reading packs
Recorded rule:
- governance docs are primarily for AI execution and retrieval
- human-facing brief should be short
- default reading pack is minimal, not full-library reading

Interpretation:
- token economy and cognitive economy are first-class design goals
- role-based retrieval matters more than giant narrative documents

### 2.5 Execution, audit, and acceptance must not collapse into one path
Recorded across role and permission docs:
- execution is not acceptance
- audit is not implementation
- field agent may not self-audit or self-accept
- supervisor should not default to implementation

Interpretation:
- anti-self-certification is a core governance law

### 2.6 High-risk work requires rollback, stop conditions, and write-back
Recorded across README / permission / discipline docs:
- high-risk work needs structured audit criteria
- stop conditions must exist
- recovery / rollback thinking must exist
- write-back target and verification responsibility must be defined

Interpretation:
- AI execution is allowed only under controlled reversibility and auditability

### 2.7 Dispatch and handoff must be explicit
Recorded rule:
- no implied ownership
- every task needs owner, boundary, write-back target, escalation path
- silence must be diagnosed, not romanticized

Interpretation:
- the system is expected to survive real operational ambiguity

### 2.8 Token cost is part of governance
Recorded rule:
- wasteful context replay is a process defect
- reuse summaries, diffs, and minimal reading packs
- spend expensive reasoning on high-value or high-risk tasks

Interpretation:
- scale must be sustainable economically, not only operationally

## 3. Structural differences

## 3.1 Role architecture
### `aion-governance`
- 8-role minimum structure
- more explicit specialization
- better for mature multi-role governance

### `three-role prototype`
- 3-role minimum structure
- Monarch / Operator Core / Audit Core
- built for low-friction AI-only startup phase

### Judgment
- old system is more detailed
- new system is more compressed and easier to automate

---

## 3.2 Historical realism vs forward compression
### `aion-governance`
- grown from real governance pain
- preserves historical nuance and specific constraints
- richer in handoff and review realism

### `three-role prototype`
- designed as a civilization kernel
- prioritizes simplicity, compressibility, and expansion path
- better for clean-slate AI-only scaling

### Judgment
- old system better preserves historical truth
- new system better supports current reinvention

---

## 3.3 Document philosophy
### `aion-governance`
- already AI-first
- routing headers
- minimal reading pack discipline
- dual-line governance model

### `three-role prototype`
- even more aggressively compressed
- oriented toward standards, templates, checklists, taxonomy
- lower-friction review pack style

### Judgment
- both fit AION's style
- new version pushes compression further

---

## 3.4 Governance style
### `aion-governance`
- more constitutional law + role law
- more explicit about fallback, hierarchy, and role collisions

### `three-role prototype`
- more operating system + workflow engine
- more focused on issue loop, templates, automation, and low token burn

### Judgment
- old version governs a richer society
- new version boots a new society faster

## 4. Which one fits AION now?

## 4.1 If AION's priority is continuity with prior governance intent
Prefer: `aion-governance`

Why:
- it contains historically recorded hard requirements
- it better reflects previous governance conversations and boundary demands

## 4.2 If AION's priority is building an AI-only organization with minimal drag
Prefer: `three-role prototype`

Why:
- it is lighter
- it is easier for AI to consume
- it is better aligned with low-human-participation workflows
- it is easier to scale with token discipline

## 4.3 Best practical answer
Best answer for current AION:
- use `aion-governance` as the historical requirement source
- use the `three-role prototype` as the current active organizational kernel

This is the best fit because AION is no longer only preserving governance; AION is trying to build a lower-friction AI civilization.

---

## 5. Migration recommendation

## 5.1 Preserve from old system
Preserve without weakening:
1. GitHub as formal center
2. Discord as short routing layer
3. no manual relay burden on Monarch
4. minimal reading packs
5. anti-self-certification rules
6. explicit dispatch / handoff
7. blocker and silence diagnosis
8. rollback / stop condition / write-back rules
9. token economy discipline

## 5.2 Compress into new system
Compress into three-role world:
- General Manager + Chief of Staff + Intelligence + Knowledge -> Operator Core internal functions
- Supervisor -> Audit Core
- Runtime Lead + Field Agent -> Operator Core execution subfunctions with explicit runtime mode

## 5.3 Do not copy directly
Do not directly preserve:
- full 8-role operational surface at day one
- all historical complexity
- every hierarchy layer if it increases friction without current value

---

## 6. Immediate next design moves

1. Create a formal migration standard from old governance to three-role governance
2. Tag historical requirements as:
   - preserve
   - compress
   - delay
   - retire
3. Produce a v0.2 three-role constitution that explicitly cites inherited laws from `aion-governance`
4. Create a review pack for external AI evaluators using both comparison and active prototype

---

## 7. Final conclusion

`aion-governance` is the stronger record of what AION historically demanded.
The three-role prototype is the better foundation for what AION is now trying to become.

Therefore the right move is not replacement.
The right move is inheritance plus compression.
