# Rule Evolution Protocol v0.1

## Purpose
Allow governance to improve without uncontrolled drift.

## Trigger conditions
A rule change may be proposed when:
- repeated friction appears
- repeated waste appears
- repeated boundary mistakes appear
- project scale changes materially
- external review identifies a serious gap

## Required proposal fields
- current rule
- observed problem
- proposed change
- expected benefit
- new risk introduced
- compatibility with inherited hard requirements

## Review sequence
1. Operator Core drafts proposal
2. Audit Core reviews for drift and overreach
3. Monarch approves constitutional-level changes

## Anti-drift rule
A new rule must justify its existence. Governance may not grow only because more text feels safer.

## Compression rule
When possible, upgrade by replacing narrative with:
- checklist
- taxonomy
- template
- decision rule
